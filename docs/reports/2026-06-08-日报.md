## 五、未修复 Bug（按严重程度）

### 5.1 已知 Bug

| # | 严重 | 描述 | 触发条件 | 复现 |
|---|---|---|---|---|
| B-001 | 中 | **OpenAI 流式响应是"模拟流式"**——一次性跑完 orchestrate()，拿到 final_answer，再按 20 字符一块推送客户端 | 用户在 Open WebUI 切换流式模式 | 直接可现：流式体验与官方 OpenAI 一致，但首字延迟 = 整次编排耗时 |
| B-002 | 中 | **OpenAI 响应 `usage` 字段全为 0** | 任何 /v1/chat/completions 调用 | 直接可现：客户端按 token 收费时会显示为 0 |
| B-003 | 中 | **不支持 OpenAI tool_calls**——Open WebUI 启用工具调用插件后会失败 | 用户在 WebUI 启用 function call 插件 | 直接可现：返回 400 或被静默忽略 |
| B-004 | 低 | **K8s manifests 未真机部署验证**——只在 18 项静态契约测试中通过 | `kubectl apply -k infra/k8s/` | 第一次真机部署可能撞到未发现的兼容性问题 |
| B-005 | 低 | **Open WebUI 用 `:main` + `imagePullPolicy: Always`** | Open WebUI 上游发版后容器自动拉新镜像 | 偶发：上游 breaking change 会让 WebUI 不可用 |
| B-006 | 低 | **Gitea token 已嵌入 `.git/config`** | 任何 `git remote -v` | 任何能读 `~/.gitconfig` 的人即可见 |

### 5.2 潜在 Bug（真机部署可能暴露）

| # | 描述 | 风险 |
|---|---|---|
| P-001 | `secretKeyRef` + `subPath: config.yaml` 同时使用——某些 K8s 版本（<1.20）会报错 | 低；现网 1.28+ 没问题 |
| P-002 | `chat_completions` endpoint 同时声明 `response_model=ChatCompletionResponse` 且可能返回 `StreamingResponse`——FastAPI 在某些版本会告警或报错 | 低；需真机跑通 |
| P-003 | Langfuse v3 init 流程在 K8s 中依赖 6 子组件按特定顺序启动——目前 `kustomization.yaml` 的 resources 顺序依赖 Kustomize 排序，可能不稳 | 中；建议加 initContainer 等待 Postgres ready |
| P-004 | Open WebUI 与 Orchestrator 间 `OPENAI_API_BASE_URL=http://orchestrator:12080/v1`——若用户改 K8s Service 端口，Open WebUI 不会感知 | 低；硬编码一次到位 |
| P-005 | Orchestrator 的 `extract_user_query` 把所有 system 消息作为前缀拼到 user query 前面——若 Open WebUI 系统提示词很长，可能撞到 4000 字上限 | 中；需 Open WebUI 端把 system prompt 控制在 500 字内 |

### 5.3 不修复 / 接受

| 描述 | 理由 |
|---|---|
| K8s Secret 明文 base64 | 仅 dev / test 阶段；生产化必走 Sealed Secrets（见 MIGRATION §4.1）|
| Langfuse 单实例 OLTP/OLAP | 自托管的妥协；企业版 Langfuse Cloud 是 HA |
| OpenAI 兼容层 `n>1` / `logprobs` 不支持 | 本项目不计划；multi-choice 没有业务价值 |
