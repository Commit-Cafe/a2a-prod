# A2A 新旧项目对比汇报

**汇报对象**：
- 旧项目：`D:\trae_Dproject3\a2a\a2a-agents\`（外层 `A2A/` 是同步下来的 [a2a-protocol.org](https://a2a-protocol.org) 官方仓库子模块，仅作参考）
- 新项目：`D:\trae_Dproject4\a2a-prod\`

**汇报时间**：2026-06-08
**汇报性质**：战略 / 架构 / 工程改进点对比 + 改进原因 + 风险点说明
**新项目当前阶段**：P0+P1（GLM Agent 跑通 A2A 协议；DeepSeek / MiniMax 占位）

---

## 0. 一句话总结

> 旧项目是一个**功能铺得很全、但自研 + 单体**的学习验证型多 Agent 系统；新项目是一个**严格控制范围、用 Google ADK + LiteLLM + a2a-sdk 工业组件**的生产化重写，强调"协议先跑通、可替换、可演进"。

两者**完全隔离**（路径、venv、端口段、Docker 网络、镜像名都错开），可以同时运行不冲突。旧项目冻结维护，所有新功能在新项目迭代。

## 1. 战略定位差异

| 维度 | 旧项目 `a2a-agents` | 新项目 `a2a-prod` |
|---|---|---|
| 版本 | v0.5.0（2026-06-04 P0-收口 / 加固完成） | v0.1.0（2026-06-05 P0-5 GLM Agent 实装） |
| 定位 | 学习 / 实验 / 功能铺全 | 生产 / 可观测 / 可演进 |
| 状态 | 冻结（仅维护） | 主开发 |
| 开发模式 | "多就是好"，先做 14 Skills + 12 Tools + 4 编排模式 | "先做根基"，P0+P1 只跑通 A2A 协议 |
| 与官方 A2A 协议仓库关系 | 同步了完整 `A2A/` 子模块（30+ 文档 + proto） | 未引入，只通过 a2a-sdk 间接对齐 |
| 自我评价（旧 README） | 代码质量 9.0/10、安全性 9.5/10 | 未做自我评分 |

**改进原因**：旧项目"功能堆叠 + 自研 + 单进程 Flask"模式在实验期有优势，但难以承担生产负载。生产化必须从"协议 + 组件可替换"开始重写。

## 2. 协议层 / Agent 框架（最核心的架构改进）

| 维度 | 旧项目 | 新项目 | 改进原因 |
|---|---|---|---|
| A2A 协议 SDK | `a2a-sdk[http-server]>=1.0.0`（部分代码仍用 `from a2a.types.a2a_pb2 import ...` 老式 protobuf 类型） | `a2a-sdk>=0.3.4,<0.4.0`（与 ADK 1.34 a2a extra 锁定对齐；Pydantic 类型系统，ADR-0005） | 旧项目混用新旧类型，proto_compat.py 打补丁；新项目用统一类型，避免协议层二次开发 |
| Agent 运行时 | 自研 `agent_executor.py`（GLM / DeepSeek / MiniMax 各手写一个 Executor） | Google ADK `LlmAgent` + `Runner`，差异点仅在 4 个属性（name / model_name / description / skills） | 自研 Executor 难以保证 A2A 状态机/事件流协议合规；ADK 原生支持 A2A 暴露，开发量下降约 60% |
| Agent 基类 | 无统一基类，三个 executor 各自定义 `SYSTEM_PROMPT` 字符串 | `agents/base_agent.py::BaseAgent` 抽象类 + `_BaseAgentExecutor` 默认实现 | 避免重复代码；强制 SPEC §2.1/§2.2 契约；子类不能重写 `run()`（防绕过） |
| LLM 适配 | 各 Agent 直接 HTTP 调用自家厂商（GLM / DeepSeek / MiniMax 各自维护 httpx 调用） | **LiteLLM Proxy**（`ghcr.io/berriai/litellm:main-stable`）统一 OpenAI 兼容接口；Agent 走 `openai/<model>` 路由 | 切换 LLM 厂商零成本；自动 retry / timeout / 流式；为多 Agent 共享 LLM 配置提供基础 |
| 自研协议补丁 | `agents/proto_compat.py`（持续打补丁兼容 a2a-sdk 升级） | 无 | 旧项目的补丁工程是技术债；新项目直接锁定稳定版本基线，避免补丁 |

**详细说明**：
- 新项目 `agents/base_agent.py` 把"差异化"压到 4 个 property（`name` / `model_name` / `description` / `skills`），GLM Agent 整个 `agent.py` 只有 86 行，差异化在 SPEC §2.2 表格里完全受控。
- 旧项目 GLM 的 `agent_executor.py` 仍能从源码里看到 `from a2a.types.a2a_pb2 import (TaskArtifactUpdateEvent, TaskState, TaskStatus, TaskStatusUpdateEvent)` 这种 a2a-sdk 1.0 之前的 protobuf 类型，说明旧项目在协议升级过程中产生了技术债。
- 新项目明确把"不自己实现协议"列为永久不做项（NORTH_STAR §3.1）。

## 3. LLM 适配层（重大架构变更）

| 维度 | 旧项目 | 新项目 |
|---|---|---|
| LLM 入口 | Agent 内直连：`httpx.AsyncClient` → `https://open.bigmodel.cn/api/coding/paas/v4` 等 | LiteLLM Proxy：`:4000` → 转发到各家 |
| 配置分散度 | 3 个 executor 各自读 `GLM_API_KEY` / `DEEPSEEK_API_KEY` / `MINIMAX_API_KEY` | 1 份 `infra/litellm/config.yaml` 集中管理；模型路由 60+ 行 yaml 即可 |
| 鉴权 | 各 Agent 自己读自己的 Key | LiteLLM master key 统一鉴权 |
| 超时/重试 | 各 Agent 写死或不一致 | LiteLLM 配置 `timeout: 60` / `retry: 3` 统一 |
| 风险 | 切厂商要改 3 处代码 + 3 处配置 | 切厂商只改 LiteLLM config.yaml，Agent 0 改动 |

**改进原因**：LLM 适配是 I/O 密集的横切关注点，应当下沉到基础设施层；让 Agent 专注协议而非 HTTP 调用细节。

## 4. 配置管理（更工程化）

| 维度 | 旧项目 | 新项目 | 改进原因 |
|---|---|---|---|
| 配置框架 | 手写 `class Config` + `os.getenv` + `_safe_get_int/float/bool` 防御性辅助函数 | `pydantic-settings`（`BaseSettings` 子类，env_file=".env.prod" 自动加载） | pydantic 自动校验 + 类型推导；少写防御代码 |

## 7. 依赖管理

| 维度 | 旧项目 | 新项目 | 改进原因 |
|---|---|---|---|
| 包管理 | pip（`requirements.txt` + `pyproject.toml`） | uv（`pyproject.toml` 标记 `[tool.uv] managed = true`） | uv 比 pip 快 5-10 倍；统一 lockfile |
| Python 版本 | `requires-python = ">=3.10"` | `requires-python = ">=3.12,<3.13"` + `.python-version` 锁定 | 严格基线；Docker 镜像 `python:3.12-slim` 与之对齐 |
| 依赖范围 | 多数用 `>=` 下限 + 不写上限（如 `a2a-sdk[http-server]>=1.0.0`） | 双开区间（如 `google-adk[a2a,extensions]>=1.34.0,<1.35.0`） | 防止 minor 升级引入 breaking |
| 核心依赖 | a2a-sdk + Flask + httpx + protobuf | google-adk + a2a-sdk + openai + httpx + fastapi + pydantic + structlog + tenacity | 新增 pydantic / structlog / tenacity（强类型 + 结构化日志 + 重试），移除非必要 protobuf |

**关键 ADR-0005**：新项目刻意锁定 ADK 1.34.x（不上 2.x），原因是"ADK 2.x 有 session schema breaking change，a2a extra 还未跟上 v1.0"。这是一个"为了稳定性暂不追新"的明确权衡。

## 10. 编排引擎（重大方向转变）

| 维度 | 旧项目 | 新项目 | 改进原因 |
|---|---|---|---|
| 编排引擎 | 自研 `agents/orchestrator/`：8 个模块（engine / agent_registry / task_analyzer / workflow_engine / negotiation_bus / skill_matcher / models / prompts） | `orchestrator/__init__.py` 占位，P2 引入 **LangGraph StateGraph** | 自研编排投入大但协议层不稳时是空中楼阁；LangGraph 已是事实标准 |
| 编排模式 | 4 种：直接路由 / 任务分解 / Agent 协商 / 动态工作流 | P2 计划：4 种模式（与旧项目对位） | 模式保留，实现替换为 LangGraph |
| 编程协作 | `agents/programming_workflow.py`（5 阶段：分析 → 选型 → 分解 → 实现 → 测试） | P2+ 引入 | 旧项目"已落地但未稳定"，新项目"待重做" |
| Singleton 模式 | `OrchestratorEngine._instance` + `threading.Lock` + `asyncio.Lock` 双重检查（旧项目 M-09 fix） | 禁止 Singleton（"不写 Singleton，用依赖注入"，CODESTYLE §4.2） | Singleton 测试困难、隐藏依赖 |
| 健康监控 | 自研 `start_health_monitor()` 线程 | 容器级 `HEALTHCHECK`（docker 自带） | 关注点分离 |

**关键取舍**：旧项目在 orchestrator 投入了大量代码（8 个模块，单例 + 双锁 + 健康监控 + 异步分析），新项目决策"全部作废"是合理的——ADK + LangGraph 已是业界成熟方案，继续自研是重复造轮子（违反用户规则 2）。但这意味着旧项目已落地的 4 编排模式经验需要在 P2 重新映射到 LangGraph。

## 14. 日志与可观测性

| 维度 | 旧项目 | 新项目 | 改进原因 |
|---|---|---|---|
| 日志库 | stdlib `logging`（`logger = logging.getLogger("...")`） | `structlog`（结构化字段） | 容器化场景下，结构化日志可直接被 Loki/ELK 采集；P4 接 Langfuse 时 0 改动 |
| 字段约束 | 无 | 每条日志 MUST 含 `request_id`（中间件注入）；禁 f-string；禁打 `api_key` / `Authorization`（CODESTYLE §6.2） | 分布式追踪基础 |
| Trace | 无 | P4 阶段 Langfuse 自托管（端口 3000） | — |
| 业务日志输出 | 混用 print | 禁 `print()`（CODESTYLE §6.1） | 容器内 stdout 由 docker 收集，需统一走 logger |

## 15. 错误处理

| 维度 | 旧项目 | 新项目 | 改进原因 |
|---|---|---|---|
| 错误类型 | Rust 风格 `Result`（`Error` / `ErrorCode` / `handle_api_error` / `handle_exception`） | 领域异常类分层：基础设施层（httpx / a2a-sdk 原生异常）→ 业务层（`AgentError` / `LLMTimeoutError` / `LLMUnauthorizedError` / `LiteLLMRoutingError`）→ 协议层（转 A2A JSON-RPC 错误码，SPEC §1.5） | 显式分层，便于按层处理；与 SPEC §1.5 错误码映射直接对应 |
| 错误码映射 | 无明确表 | SPEC §1.5 全表：LLM 超时 → -32603 / Key 无效 → -32001 / LiteLLM 路由失败 → -32602 / 解析错误 → -32700 / 方法不存在 → -32601 | A2A 协议合规，跨 Agent 行为一致 |
| 禁项 | 无 | CODESTYLE §5.2：禁裸 `except:` / 静默 `except Exception: pass` / 循环内 try/except 不记日志 | 防吞错 |

## 16. 范围控制（最重要的思维转变）

**旧项目的"范围蔓延"迹象**（从 README 看）：
- v0.2.5 引入 14 Skills + 12 Tools（"集成 14 个 Matt Pocock Skills"）
- v0.3.0 加 4 编排模式（"智能单 Agent 路由 / 多 Agent 任务分解 / Agent 间自主协商 / 动态工作流"）
- README §"路线图" 列了"短期 / 中期 / 长期"共 15+ 项
- 19+ 个 .md 设计文档（含"OPENCODE_A2A_INTEGRATION_DESIGN" 3 个 Provider × 2 版）

**新项目的"范围控制"**（NORTH_STAR §3）：
- 永远不做：不自己实现 A2A 协议 / 不自己实现 LLM 适配 / 不硬编码 API Key / 不做与旧项目双向同步
- 当前阶段不做：LangGraph 编排（P2）/ MCP（P3）/ Langfuse（P4）/ Open WebUI（P5）/ K8s（P6）
- 退出条件：连续 3 次跑不通 → 记录 ADR → 退到备选方案

**改进原因**：旧项目实验期"先做出来"思路导致代码量大、技术债深（如 3 个 executor 各写各的 SYSTEM_PROMPT 字符串、proto_compat 持续打补丁）。新项目明确"先做根基，根基不稳上层是空中楼阁"（ADR-0003）。

## 17. 关键 ADR 决策摘要（新项目特色）

| ADR | 决策 | 关键理由 |
|---|---|---|
| ADR-0001 | 采用 Google ADK + LiteLLM 双层架构 | ADK 原生支持 A2A 暴露，零样板；LiteLLM 已支持 100+ LLM |
| ADR-0002 | 与原 a2a-agents 完全隔离 | 用户明确要求"互不打扰" |
| ADR-0003 | 首发里程碑选 P0+P1 | 协议层是根基，先证明国产三模型能标准 A2A 协作 |
| ADR-0004 | 前端选 Open WebUI 而非自研 Vue | UI 完成度远超自研；省 80% 前端工作量 |
| ADR-0005 | 依赖版本基线：ADK 1.34.x + a2a-sdk 0.3.x | ADK 2.x 有 breaking change，a2a extra 未跟上 v1.0 |
| ADR-0006 | SPEC.md 作为项目规范单一事实源 | 协议/接口契约集中化；review 标准统一 |

## 20. 结论

### 改进点（必做，已做或在做）
1. **协议层从自研 → 官方 SDK**（ADK + a2a-sdk）
2. **LLM 适配从散在各 Agent → 统一 LiteLLM Proxy**
3. **配置从手写 → pydantic-settings**
4. **日志从 stdlib → structlog（结构化）**
5. **文档从散在多份 → 6 份核心文档 + 仲裁顺序**
6. **代码规范从弱约束 → mypy --strict + 长度上限 + 命名约定**
7. **容器化从单 Dockerfile → 共享 Dockerfile + docker-compose + 健康检查 + 非 root**
8. **依赖从 pip + 单边范围 → uv + 双开区间 + .python-version**
9. **范围控制从"多多益善" → "必做 + 不做 + 退出条件"清单**
10. **编排引擎从自研 8 模块 → 计划替换为 LangGraph**（P2 待实施）

### 风险点（需用户决策）
- P0+P1 阶段对终端用户不可用（无 UI、无编排、无鉴权）
- 失去旧项目 12 工具 + 14 Skills + 4 编排模式 + Vue 前端的已实现能力（合理但需明确告知）
- ADK 1.34.x 版本锁定有保质期（ADR-0005 退出条件）
- 范围控制纪律需团队养成，否则 spec-violation 机制形同虚设

### 借鉴清单（供 P2~P6 阶段参考）
旧项目的 4 编排模式分类、5 阶段编程工作流、12 工具安全沙箱、Agent 元数据 SSOT 思想、`check_api_keys.py` 预检脚本等经验可直接在新项目对应阶段复用，避免完全从零开始（符合用户规则 2）。
